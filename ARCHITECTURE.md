# Архитектура

## Обзор

Клиент-серверное веб-приложение в локальной сети кинотеатра (интернет доступен).

```
┌─────────────┐     ┌─────────────┐     ┌──────────────────┐
│  Касса      │     │  Админка    │     │ Дисплей гостей   │
│ (browser)   │     │ (browser)   │     │ (browser/киоск)  │
└──────┬──────┘     └──────┬──────┘     └────────┬─────────┘
       │ REST + WS         │ REST              WS │ (только чтение)
       └───────────────────┴──────────────────────┘
                           │
                  ┌────────▼────────┐
                  │   NestJS API    │  REST + Socket.IO
                  │  (apps/api)     │
                  └────────┬────────┘
                           │ Prisma
                  ┌────────▼────────┐
                  │   PostgreSQL    │
                  └─────────────────┘
```

## Конфигурация зала (важно)

Зал асимметричный: 6 рядов, проход посередине делит каждый ряд на левый блок (места 1–4)
и правый блок. В рядах 1–5 правый блок — места 5–8; в ряду 6 правый блок — места 5–9.
Итого **49 мест**.

| Ряд | Левый блок | Правый блок | Всего |
|-----|-----------|-------------|-------|
| 1 | 1–4 | 5–8 | 8 |
| 2 | 1–4 | 5–8 | 8 |
| 3 | 1–4 | 5–8 | 8 |
| 4 | 1–4 | 5–8 | 8 |
| 5 | 1–4 | 5–8 | 8 |
| 6 | 1–4 | 5–9 | 9 |

Конфигурация хранится в БД явно (таблицы `Hall` → `Seat`), а не вычисляется формулой —
это закладывает расширение на залы с другой раскладкой (вторая очередь ТЗ).
Каноничная раскладка для сидов — в `packages/shared` (`HALL_1_LAYOUT`).

## Состояние места (статусы)

| Статус | Цвет (ТЗ) | Описание |
|--------|-----------|----------|
| `FREE` | красный | свободно, можно продать |
| `SOLD` | серый | продано (занято) |
| `SELECTED` | зелёный | выбрано в текущем заказе (только на кассе, не в БД) |
| `BOOKED` | жёлтый | забронировано без оплаты, с авто-сбросом |

`SELECTED` — это клиентское состояние кассы, в БД его нет. В БД место в рамках сеанса
имеет статус `FREE` / `SOLD` / `BOOKED` (через таблицы `Ticket` / `Booking`).

## Real-time (Socket.IO)

Главное требование ТЗ: продал место → у гостя сразу серое, без перезагрузки.

- Комнаты по сеансу: `session:{sessionId}`.
- При продаже/брони/возврате API эмитит `seat:update { sessionId, seatId, status }`.
- Дисплей гостей подписан на комнату текущего сеанса и обновляет схему мгновенно.
- Касса тоже слушает комнату — защита от двойной продажи одного места двумя кассирами
  (плюс транзакция + уникальный constraint в БД как источник истины).

## Защита от двойной продажи

Уникальный constraint `(sessionId, seatId)` на проданных/забронированных местах +
транзакция Prisma. Если два кассира кликнули одно место — второй получит ошибку,
а не «успех».

## i18n

`react-i18next`, языки `ru` / `ky`, переключатель в шапке. Все строки — в ресурсах,
не хардкодом.

## Дальнейшие этапы (вторая очередь)

Архитектура закладывает, но первый этап не реализует: несколько залов, сканирование
билетов на входе, программа лояльности, онлайн-продажа, фискальный регистратор,
мобильное приложение админа.
